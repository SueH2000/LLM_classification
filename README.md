# LLM_classification

A reproducible pipeline to classify data provenance in biomedical papers:

- **Primary**: the paper mainly uses data generated by the authors.
- **Reuse**: the paper mainly reuses public/external datasets.
- **Unclear**: not enough evidence to make a safe decision.

---

## Why this repo moved from notebook to script (beginner-friendly explanation)

The notebook prototype was useful for exploration, but production use usually needs a script because scripts are easier to reproduce and audit.

Common notebook problems:
1. Cell execution order can drift (results may change unintentionally).
2. Parameters are scattered across cells.
3. Logic can be duplicated in multiple cells and become hard to maintain.

The script `rag_lightweight_pipeline.py` fixes this by:
- centralizing parameters in one config object,
- supporting CLI arguments for controlled experiments,
- saving `run_config.json` so each run is traceable.

## Pipeline overview (step by step)

### Step 1) Read JSONL and collect text
The script reads common fields (`title`, `abstract`, `full_text`, `body`, `text`) and concatenates them.

**Why:** provenance clues are often spread across different sections.

### Step 2) Build evidence windows
- Detect accession IDs (GSE/GSM/SRP/SRR/ENA/...)
- Extract nearby context windows around each ID
- If no accession windows are found, fallback to provenance keywords (e.g., "downloaded", "publicly available")

**Why:** data-origin clues are often close to accession mentions or in data-availability text.

### Step 3) Heuristic triage
Run cheap rule-based classification first.

**Why:** reduce LLM calls and cost. Obvious cases are solved quickly; uncertain cases can be escalated.

### Step 4) Lightweight RAG (no heavy fine-tuning)
Build a TF-IDF retriever from your labeled CSV and retrieve top-k similar examples for each target paper. Those examples are inserted into the prompt.

**Why:** this improves consistency without training a new model.

### Step 5) Save outputs for reproducibility
- `predictions_rag.csv`: row-level prediction outputs
- `run_config.json`: exact settings used in this run

**Why:** you can re-run or compare experiments later.

## How to run with a better local LLM model

Short answer: **Yes, running RAG locally with a stronger model usually helps quality**.

RAG improves context; model quality determines reasoning strength. You generally want both.

### 1) Start Ollama and pull a stronger model

```bash
ollama serve
ollama pull llama3.1:8b
```

(Use another model if your hardware allows, e.g. 70b variants.)

### 2) Run pipeline with that model

```bash
python rag_lightweight_pipeline.py \
  --jsonl-path pmc_gse_articles_clean.jsonl \
  --labeled-csv-path manual_ground_truth_with_GSE_links_REFRESHED.csv \
  --llm-mode unclear_only \
  --ollama-model llama3.1:8b
```

### 3) Increase retrieval support when dataset is larger

```bash
python rag_lightweight_pipeline.py \
  --jsonl-path pmc_gse_articles_clean.jsonl \
  --labeled-csv-path manual_ground_truth_with_GSE_links_REFRESHED.csv \
  --llm-mode unclear_only \
  --ollama-model llama3.1:8b \
  --rag-top-k 5 \
  --rag-max-examples 200
```

## Key parameters

- `--jsonl-path`: input JSONL (required)
- `--labeled-csv-path`: labeled examples CSV (`ground_truth` or `human_label`)
- `--llm-mode`: `off | unclear_only | all`
- `--seed`: deterministic seed (default `42`)
- `--rag-top-k`: number of retrieved examples per query
- `--rag-max-examples`: cap of labeled rows used to build retriever index
- `--ollama-model`: local model name served by Ollama
- `--ollama-url`: Ollama endpoint URL (default localhost)
- `--phrase-hints-path`: optional `mined_phrases.json` used to add class phrase hints into the RAG prompt
- `--phrase-hints-per-class`: number of mined phrases per class injected into prompt

### `ModuleNotFoundError: numpy`
Install dependencies:

```bash
pip install -U -r requirements.txt
```

### Ollama connection fails
Check:
- `ollama serve` is running
- `--ollama-url` is correct
- model in `--ollama-model` exists locally

### Too many `Unclear` predictions
Try this order:
1. Increase `--rag-max-examples`
2. Improve label quality in CSV
3. Use `--llm-mode unclear_only` instead of `off`
4. Ensure JSONL has enough `full_text`

## Systematic upgrade: from manual keywords to data-driven evidence modeling

You asked for a more systematic pipeline. This repo now includes `evidence_modeling.py` to do exactly that.

### What this script does

1. **Auto phrase mining from labeled CSV**
   - Mines top-k discriminative n-grams for `Primary` and `Reuse` from the training split.
   - Output: `mined_phrases.json`

2. **Convert mined phrases into pattern templates**
   - Converts phrases into whitespace-tolerant regex templates.
   - Uses template counts to classify each evidence text.

3. **Train sentence-level linear model on evidence text**
   - Uses TF-IDF features + Logistic Regression.
   - Trained on train split only (no leakage).

4. **Compare all versions on the same split**
   - `static_rules` (your manual-rule style baseline)
   - `mined_template_rules` (data-driven template rules)
   - `linear_model` (evidence-based supervised model)
   - Output: `model_comparison.csv`


### Matching rule (important)

For the systematic script (`evidence_modeling.py`), labels and articles are matched by `paper_id`:

- CSV provides labels (`paper_id` + `ground_truth`/`human_label`)
- JSONL provides article text (same `paper_id`)
- The script performs an **inner join** on `paper_id`

This prevents accidental label/text mismatch and is safer than relying on row order.

### Why this is better for beginners and maintenance

- You keep a clear baseline (`static_rules`).
- You add data-driven phrase discovery (less guesswork than manual keyword updates).
- You add a supervised model for stronger generalization.
- You compare everything on one fixed split, so improvements are measurable.

### Run command

```bash
python evidence_modeling.py \
  --labeled-csv-path manual_ground_truth_with_GSE_links_REFRESHED.csv \
  --jsonl-path pmc_gse_articles_clean.jsonl \
  --out-dir outputs_evidence_modeling \
  --top-k-phrases 40 \
  --test-size 0.2 \
  --random-state 42
```

### Main outputs

- `outputs_evidence_modeling/model_comparison.csv`
- `outputs_evidence_modeling/test_predictions.csv`
- `outputs_evidence_modeling/mined_phrases.json`
- `outputs_evidence_modeling/linear_model_report.txt`

Tip: if `mined_template_rules` beats `static_rules`, you can gradually replace hand-written patterns with mined templates.


## How to use evidence-modeling findings to improve RAG prompt and tuning

Great question: the outputs from `evidence_modeling.py` should directly feed your RAG stage.

### 1) Generate mined phrases first

```bash
python evidence_modeling.py \
  --labeled-csv-path manual_ground_truth_with_GSE_links_REFRESHED.csv \
  --jsonl-path pmc_gse_articles_clean.jsonl \
  --out-dir outputs_evidence_modeling
```

This creates `outputs_evidence_modeling/mined_phrases.json`.

### 2) Use mined phrases as prompt hints in RAG

```bash
python rag_lightweight_pipeline.py \
  --jsonl-path pmc_gse_articles_clean.jsonl \
  --labeled-csv-path manual_ground_truth_with_GSE_links_REFRESHED.csv \
  --llm-mode unclear_only \
  --ollama-model llama3.1:8b \
  --phrase-hints-path outputs_evidence_modeling/mined_phrases.json \
  --phrase-hints-per-class 8
```

What this does:
- keeps your normal retrieved examples (RAG retrieval)
- additionally injects mined class phrases as **soft cues** in prompt
- gives the LLM both case-based context and class-level phrase priors

### 3) Tuning strategy (practical)

- Start with `--phrase-hints-per-class 5~10`
- If prompt becomes noisy, reduce hint count before changing model
- Compare with/without `--phrase-hints-path` on same split
- Keep `--llm-mode unclear_only` first for stable cost and easier debugging
